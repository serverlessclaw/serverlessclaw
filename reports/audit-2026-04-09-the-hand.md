# Audit Report: Silo 2 - The Hand (Agency & Skill Mastery)

**Date**: 2026-04-09
**Auditor**: Gemini CLI
**Topic**: Agent Intent to Tool Execution & MCP Integration

---

## 🔍 Overview

This audit focused on the **Hand Silo**, specifically the boundary between agent reasoning and tool execution. We investigated the Model Context Protocol (MCP) bridge, client management, tool mapping, and the core agent execution loop.

## 🚨 Critical Findings (P0)

### 1. Workspace Isolation Breach in MCP Stdio Transport
*   **Silo**: Spine ↔ Hand
*   **Description**: The `MCPClientManager` caches `Client` instances in a static `Map` shared across warm Lambda starts. Local MCP servers (e.g., `filesystem`, `git`) use `StdioClientTransport`, which roots the server process in the current working directory (`cwd`) at the time of the *first* connection.
*   **Expected**: Each task invocation (especially `Coder` agent tasks) operates in a unique temporary workspace (`/tmp/workspace-XYZ`). The `filesystem` tool should always point to the *current* task's workspace.
*   **Actual**: In warm starts, if a previous invocation already initialized the `filesystem` server in `/tmp/workspace-A`, a subsequent invocation in `/tmp/workspace-B` will receive the cached client still rooted in `/tmp/workspace-A`.
*   **Impact**: Agents will read from and write to the wrong workspace. This leads to data leakage between tasks, failed builds, and potential security risks if an agent modifies a previous task's files instead of the current one.
*   **Severity**: **P0**
*   **Recommended Action**: Update `MCPClientManager.connect` to include `process.cwd()` in the cache key or detect `cwd` changes for local servers and force a restart.

## ⚠️ High-Priority Issues (P1)

### 2. Missing Runtime Validation for MCP Tools
*   **Silo**: Hand
*   **Description**: `MCPToolMapper` successfully extracts JSON Schema from MCP servers but maps it only to `ITool.parameters`. It sets `ITool.argSchema` (the Zod-based validation layer) to `z.any()`.
*   **Impact**: `ToolExecutor` performs no runtime validation on MCP tool arguments. Invalid arguments generated by the LLM are passed directly to the MCP server, resulting in downstream failures that are harder to debug and recover from compared to failing at the agent boundary.
*   **Severity**: **P1**
*   **Recommended Action**: Implement a utility (e.g., `jsonSchemaToZod`) to dynamically generate Zod schemas from the MCP tool's `inputSchema` and assign it to `argSchema`.

## 📉 Architectural Debt (P2)

### 3. Potential Serialization Failure in `LambdaInvokeTransport`
*   **Silo**: Spine ↔ Hand
*   **Description**: In `LambdaInvokeTransport.send`, the code passes `payload.body` directly to `this.onmessage`. In typical AWS Lambda proxy responses, `body` is a stringified JSON. The MCP SDK `Client` expects a parsed `JSONRPCMessage` object.
*   **Severity**: **P2**
*   **Recommended Action**: Verify if `payload.body` is a string and `JSON.parse` it before calling `onmessage`. Add explicit tests for `LambdaInvokeTransport`.

### 4. Inconsistent MCP Tool Caching Logic
*   **Silo**: Brain ↔ Hand
*   **Description**: `MCPBridge.getToolsFromServer` uses a per-server cache key (`mcp_tools_cache_${serverName}`), while the global `getCachedTools` helper looks for a single `mcp_tools_cache` key which likely doesn't exist or contains different data.
*   **Severity**: **P2**
*   **Recommended Action**: Standardize the caching strategy for tool definitions.

## 💡 Observations & Improvements (P3)

### 5. Brittle Filesystem Security Heuristics
*   **Description**: `checkFileSecurity` in `MCPToolMapper` relies on a hardcoded list of argument names (`path`, `path_to_file`, `file_path`) to identify file targets for protection checks.
*   **Improvement**: Centralize security checks or use a more robust way to identify "path-like" arguments across all tools.

### 6. MCP Tool Mapping Duplication
*   **Description**: `mapTools` and `mapCachedTools` in `MCPToolMapper` share ~95% of their logic.
*   **Improvement**: Refactor into a single private mapping method to ensure consistency in security and parameter injection.

## 🛠️ Verification Results

*   **Static Probes**: Reviewed `core/lib/mcp.ts`, `core/lib/mcp/client-manager.ts`, `core/lib/mcp/tool-mapper.ts`, `core/lib/agent.ts`, and `core/lib/agent/tool-executor.ts`.
*   **Logic Probes**: Identified state leakage in `static` map usage within a warm-start environment (Lambda).
*   **Conclusion**: The system has a robust foundation for multi-transport tool execution, but the "Stateless Core" principle (Principle 1) is currently violated by the `MCPClientManager` caching strategy, leading to a critical workspace isolation bug.
